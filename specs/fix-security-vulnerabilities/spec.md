# Feature: Remediate Security Vulnerabilities

## Description

This feature addresses critical, high, medium, and low-severity security findings in the `authentication-service` and `frontend` applications. The goal is to strengthen the security posture by fixing vulnerabilities related to CSRF, CORS, JWT key management, and security filter configurations.

## Clarifications
### Session 2026-06-04
- Q: CSRF strategy? → A: Enable CSRF with double-submit token (XSRF-TOKEN cookie + X-XSRF-TOKEN header).
- Q: CORS error-path handling? → A: Disable CORS headers on error paths entirely.
- Q: CORS allowlist source? → A: Use `authentication.cors.allowed-origin-patterns` from `application.yaml`.
- Q: JWT signing key strategy? → A: Load keys from a file/volume generated by the script; no silent generation.
- Q: Startup behavior when JWT key missing/unconfigured? → A: Fail startup with a clear error.
- Q: CSRF token cookie attributes? → A: HttpOnly=false, SameSite=Lax, Secure=yes, Domain=existing cookie domain, Path=/.
- Q: Security findings scope? → A: Include medium and low findings.
- Q: Wildcard matching for filter exclusions? → A: Use Ant-style path matching (e.g., /api/public/**).
- Q: Filter exclusion scope? → A: Allowlist only; default deny.

## Security Findings to Address

1.  **Critical - CSRF disabled with cookie auth**: Cross-Site Request Forgery (CSRF) protection is currently disabled, but the application uses cookie-based authentication. This creates a significant vulnerability where an attacker could trick a user into performing unintended actions. **Chosen strategy**: enable CSRF protection using a double-submit token (`XSRF-TOKEN` cookie + `X-XSRF-TOKEN` header) with cookie attributes set to HttpOnly=false, SameSite=Lax, Secure=yes, and explicit domain/path (existing cookie domain, `/`).
2.  **Critical - Error-path CORS reflects any origin**: The Cross-Origin Resource Sharing (CORS) policy on error paths is too permissive, reflecting any origin. This could allow malicious websites to access error details, potentially leaking sensitive information. **Chosen strategy**: use the `authentication.cors.allowed-origin-patterns` allowlist from `application.yaml` for normal paths and disable CORS headers on error paths entirely.
3.  **High - Ephemeral JWT key with no warning/guard**: JSON Web Token (JWT) signing keys are generated at startup (ephemeral) and are not persisted. This means all user sessions are invalidated upon application restart, leading to a poor user experience and potential security issues if not handled correctly. **Chosen strategy**: load keys from a file/volume generated by the script, do not auto-generate, and fail startup with a clear error when missing or unconfigured.
4.  **High - Filter path exclusion ignores wildcards**: The security filter path exclusion mechanism does not correctly handle wildcard patterns (e.g., `/api/public/**`). This could lead to security filters not being applied to intended paths, leaving them unprotected. **Chosen strategy**: use Ant-style wildcard matching and restrict exclusions to an explicit allowlist (default deny).
5.  **Medium/Low - Remaining scanner findings**: Address all medium and low severity findings identified for the `authentication-service` and `frontend` applications within this feature scope.
